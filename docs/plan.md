# 実装計画: そこねこ(Phase 2: Plan)

> Status: **Approved(2026-07-12・タスク分解への進行指示をもって承認)** / 最終更新: 2026-07-12
> 対象仕様: `docs/spec.md`(Approved 2026-07-12)
> タスク分解: `docs/tasks.md` / デザインガイド: `docs/design.md`

## 方針

- 「縦に薄く通す」進め方をとる。まず認証〜価格記録〜底値一覧の最小フローを
  エミュレータ上で動くところまで作り、その後に比較・編集削除・オフライン・PWA を積む
- 各フェーズの終わりに検証チェックポイントを置き、動作確認してから次へ進む
- Firestore セキュリティルールは「後回しにすると全開放のまま進みがち」なので、
  データアクセス実装(フェーズ 3)と同時に書き、ルールテストも同時に用意する

## フェーズ分割

### フェーズ 0: プロジェクト基盤

- Vite + React 19 + TypeScript プロジェクト生成、Tailwind CSS v4 導入
- ESLint + Prettier、Vitest + React Testing Library のセットアップ
- Firebase プロジェクト作成(コンソール)、Firebase SDK 初期化、Emulator Suite 設定
- `npm run dev / build / test / lint / emulators` が全部動く状態にする

**検証**: 空アプリがエミュレータ接続込みで起動し、サンプルテストと lint が通る

### フェーズ 1: 認証と book 初期化

- Firebase Auth(Google + メール/パスワード + パスワードリセット)
- ログイン画面、認証ガード(未ログイン → ログイン画面へ)
- 初回ログイン時の book 初期化(getDoc 存在確認 + create 専用トランザクション、
  決定的 ID でカテゴリシード)
- book / 認証まわりのセキュリティルール(create 時 `bookId == uid` 等)とルールテスト

**検証**: エミュレータで新規登録 → book とシードカテゴリが 1 回だけ作られる。
2 回目ログインで上書きされないことをテストで確認

### フェーズ 2: マスタ管理(カテゴリ・店舗・商品)

- カテゴリ一覧・追加・編集(baseUnit 選択)、店舗の追加・編集、商品の登録・編集
- 参照整合性: 参照中の店舗・カテゴリの削除ブロック(件数提示)、
  商品削除時の priceRecords 分割バッチ削除
- 単位換算ロジック(`lib/units.ts`: kg→g、L→ml)と単体テスト

**検証**: マスタの CRUD が動き、単位換算の単体テストが通る

### フェーズ 3: 価格記録と底値一覧(アプリの心臓部)

- 価格記録フォーム(商品・店舗のその場追加、特売フラグ、総量入力ヒント)
- 底値算出ロジック(直近 N ヶ月、特売込み/通常のみ)と単体テスト
- ホーム(底値一覧: 検索・カテゴリ絞り込み・特売バッジ)
- 商品詳細(履歴一覧、記録の編集・削除、店舗別底値、特売込み/通常のみ併記)
- priceRecords のセキュリティルール(正数検証)とルールテスト

**検証**: 記録 → ホームに底値が反映される一連のフローがエミュレータで動く。
底値算出・単価計算のテストが通る

### フェーズ 4: カテゴリ内比較と設定

- カテゴリ内単価ランキング画面
- 設定画面(カテゴリ・店舗管理への導線、底値の対象期間 N の変更、ログアウト)

**検証**: 異なる内容量・単位の商品がランキングで正しく並ぶ

### フェーズ 5: オフラインと PWA

- Firestore 永続化(`persistentLocalCache` + `persistentMultipleTabManager`)
- vite-plugin-pwa(manifest、`registerType: 'prompt'` の更新プロンプト UI)
- 孤児レコードのフォールバック表示(「(不明な店舗)」等)の確認

**検証**: 機内モードで閲覧・記録 → 復帰で同期(手動 E2E チェックリスト実施)

### フェーズ 6: 本番デプロイ

- Firebase プロジェクトの本番設定(Auth 承認済みドメイン等)
- `firebase deploy`(Hosting + ルール + インデックス)
- スマホ実機で PWA インストール・全 Success Criteria の最終確認

**検証**: `docs/spec.md` の Success Criteria 全項目にチェックが付く

## 依存関係と並行性

- フェーズ 0 → 1 → 3 は直列(それぞれ前段に依存)
- フェーズ 2 は 1 の完了後、3 と一部並行可(フォームのその場追加は 2 の API を使う)
- フェーズ 5 の PWA 設定はフェーズ 0 直後から入れることも可能だが、
  SW キャッシュがあると開発中のデバッグが紛らわしいため、意図的に終盤に置く

## リスクと対策

| リスク | 対策 |
|---|---|
| Tailwind v4 / React 19 など新しめのバージョンの組み合わせ問題 | フェーズ 0 で最小構成の動作確認を先に済ませる。問題があれば安定版に落とす(Ask first) |
| セキュリティルールの書き漏れ | ルールはデータアクセス実装と同フェーズで書く。ルールテストを Success Criteria に含めている |
| オフライン同期の競合(同一記録の同時編集) | MVP は個人利用のため「後勝ち」で許容。仕様の孤児フォールバックで表示クラッシュは防ぐ |
| Firestore 無料枠超過 | 個人利用の規模では実質起きない。従量課金機能の追加は Ask first 済み |
| 店頭での入力が遅いと使われなくなる | フェーズ 3 で「3 タップ + 数値入力」目標を満たすか実機確認し、満たさなければ入力 UI を優先改善 |
