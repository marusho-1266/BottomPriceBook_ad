## ヒアリング結果(2026-07-18 確定)

1. **スコープ**: Issue 本文はプライバシーポリシー+問い合わせ窓口のみだが、タイトル通り
   **利用規約とプライバシーポリシーの両方**を掲載する(一般公開時に揃っている方が安心なため)
2. **問い合わせ窓口**: **Google フォーム**を採用。無料・実装不要でリンクを張るだけで済み、
   個人メールアドレスを公開せずに済む
3. **文面の用意**: **Claude がドラフトを作成**し、ユーザーがレビュー・修正して確定する。
   収集情報(メールアドレス、Firebase Auth / Firestore、Sentry・GA4 の利用)を踏まえて起草する。
   **法的助言ではない**点に留意し、最終確認はユーザーが行う
4. **実装方式**: **アプリ内ページ**(React ルート `/terms` / `/privacy`)として実装。
   ログイン前でも閲覧できるよう構成変更が必要だが、デザインを既存 UI と統一できる

## 前提(仮定 — 誤りがあれば指摘してください)

1. 現状 `LoginScreen` は `BrowserRouter` の**外**(未ログイン分岐)にあり、URL によるページ遷移が
   できない(`src/App.tsx` の `Gate` 参照)。本 Issue で `BrowserRouter` を `App` 直下に移動し、
   `/terms` / `/privacy` を**認証不要の公開ルート**として定義する構成変更を行う
2. 上記の構成変更後も、既存のログイン後ルーティング(`AppShell` 配下・`/join`)の挙動は
   変えない(リグレッションなし)
3. Google フォームの作成(質問項目: お名前(任意)・返信先メールアドレス(任意)・お問い合わせ内容)は
   **ユーザーの手動作業**。フォーム URL は公開情報のためソースコードに直接記載してよい
   (環境変数にはしない)
4. ポリシーに記載する収集・利用情報は以下(現行実装ベース):
   - **アカウント情報**: メールアドレス、Google アカウントの表示名(Firebase Authentication)
   - **ユーザーが入力したデータ**: 商品・価格記録・店舗・カテゴリ等(Cloud Firestore)
   - **利用状況**: Firebase Analytics(GA4)による自動収集イベント+カスタムイベント
     (個人情報・入力値は含めない。Issue #18 の方針)
   - **エラー情報**: Sentry(PII はマスク済み。Issue #18 の方針)
   - **Cookie / ローカルストレージ**: 認証セッション維持・Analytics のために利用
5. Cookie 同意バナーは本 Issue でも実装しない(Issue #18 のヒアリングで見送り済み。
   ポリシー本文に Cookie 等の利用を明記することで足りるとする)
6. 事業者表記は個人開発のため氏名の公開はせず「そこねこ運営者」等の表記とする
   (正式な表記は Open Questions で確定)
7. 特定商取引法に基づく表記は不要(無料アプリで販売行為がないため)

## Objective

### 何を作るか

- 利用規約ページ(`/terms`)とプライバシーポリシーページ(`/privacy`)を、
  **ログイン前でも閲覧できる**アプリ内ページとして追加する
- ログイン画面のフッターに両ページ+問い合わせフォームへのリンクを設置する
- ログイン後の設定画面からも両ページ+問い合わせフォームへ遷移できるようにする
- 問い合わせ窓口として Google フォームを用意し、ポリシー内・各リンク箇所から案内する

### 背景・課題

メールアドレスという個人情報を収集するため、一般公開時にはプライバシーポリシーの掲示
(ログイン画面からのリンク)と問い合わせ窓口が必要(Issue #14)。一般公開の前提条件の一つ。

### ユーザーストーリー

- 登録を検討しているユーザーとして、**登録前に**どんな情報が収集・利用されるかを確認したい
- 利用中のユーザーとして、困ったときに運営へ問い合わせる窓口がほしい
- 運営者として、個人のメールアドレスを公開せずに問い合わせを受け付けたい

## 実装内容

### 1. ルーティング構成変更(公開ルートの導入)

`src/App.tsx` を変更し、`BrowserRouter` を `App` 直下(認証状態の分岐より外)に移動する。

```
<AuthProvider>
  <BrowserRouter>
    <Routes>
      <Route path="terms" element={<TermsPage />} />      ← 公開(認証不要)
      <Route path="privacy" element={<PrivacyPage />} />  ← 公開(認証不要)
      <Route path="*" element={<Gate />} />               ← 既存の認証ゲート
    </Routes>
  </BrowserRouter>
</AuthProvider>
```

- `Gate` 内の `BrowserRouter` は撤去し、既存の `Routes`(AppShell 配下・`/join`)は
  `Gate` 内の入れ子 `Routes` として維持する
- 未ログインで `/terms` / `/privacy` を開くとログイン画面ではなくポリシーページが表示される
- ログイン済みで開いた場合も同じページを表示する(認証状態に依存しない)

### 2. ポリシーページ(TermsPage / PrivacyPage)

| 項目 | 内容 |
|---|---|
| 配置 | `src/features/legal/TermsPage.tsx` / `PrivacyPage.tsx` |
| 文面 | `src/features/legal/` 配下に本文コンポーネント(または定数)として保持。Markdown レンダラ等の新規依存は追加しない |
| デザイン | 既存トーン(bg-cream・rounded カード・`SubPageHeader` 相当の戻るヘッダー)に合わせる。`SubPageHeader` が AppShell/認証前提でなければ再利用、前提があれば軽量ヘッダーを新設 |
| 戻る導線 | 未ログイン時はログイン画面(`/`)へ、ログイン時は遷移元(history back)へ戻れること |
| 問い合わせ | ポリシー本文の「お問い合わせ」節に Google フォームへのリンク(`target="_blank"` + `rel="noopener noreferrer"`)を記載 |

### 3. 文面ドラフト(Claude 作成 → ユーザー確定)

**プライバシーポリシー**に含める節(前提 4 の収集情報を反映):

1. 事業者(運営者)表記
2. 収集する情報(アカウント情報・入力データ・利用状況・エラー情報)
3. 利用目的(サービス提供・品質改善・不具合対応)
4. 第三者提供(原則なし。Google(Firebase/GA4)・Sentry への処理委託を明記)
5. Cookie 等の利用
6. データの保存期間と削除(退会機能 Issue #13 によるユーザー主導の削除を明記)
7. ポリシーの変更手続き
8. お問い合わせ窓口(Google フォーム)
9. 制定日・改定日

**利用規約**に含める節:

1. 適用範囲・同意
2. アカウント(登録・管理責任)
3. 禁止事項(不正アクセス・他ユーザーへの迷惑行為等)
4. サービスの提供・変更・中断・終了(個人開発の無料サービスである旨)
5. 免責(価格情報の正確性を保証しない等)
6. 知的財産
7. 規約の変更
8. 準拠法・管轄

### 4. リンク設置

| 場所 | 内容 |
|---|---|
| ログイン画面(`LoginScreen`)フッター | 「利用規約」「プライバシーポリシー」「お問い合わせ」の 3 リンクを画面下部に追加。規約・ポリシーは `<Link>`(公開ルート)、お問い合わせは Google フォームへの外部リンク |
| 設定画面(`SettingsPage`) | 既存のメニュー形式に合わせて「利用規約」「プライバシーポリシー」「お問い合わせ」の行を追加 |

## Boundaries(本 Issue 固有)

- **Always**
  - ポリシー・規約ページは認証なしで閲覧できる状態を維持する
  - 外部リンク(Google フォーム)には `rel="noopener noreferrer"` を付ける
  - ルーティング変更時は既存テストを全て通す
- **Ask first**
  - 文面ドラフトの確定(ユーザーのレビュー・修正を経てから実装に反映)
  - 事業者表記・制定日など文面の可変項目の決定
  - 新規依存パッケージの追加(現時点では不要の想定)
- **Never**
  - 文面に個人の氏名・住所・個人メールアドレスを無断で記載しない
  - 実装と乖離した記載をしない(収集していない情報を「収集する」と書かない、逆も)
