# Spec: Issue #20 データエクスポート(CSV 等)機能

> Status: **Draft(ヒアリング確定・実装前)** / 作成: 2026-07-20
> 対象 Issue: [#20 データエクスポート(CSV 等)機能](https://github.com/marusho-1266/BottomPriceBook_ad/issues/20)
> 親仕様: `docs/spec.md`

## ヒアリング結果(2026-07-20 確定)

1. **対象データ範囲**: 価格記録(`PriceRecord`)のみ。商品・カテゴリ・店舗のマスタは別シート/別 CSV としては出力しない
2. **起動場所**: 設定画面(`SettingsPage.tsx`)に「データをエクスポート」ボタンを常設(退会フローに限らず、いつでも使える)
3. **文字コード**: UTF-8(BOM 付き)。Excel(Windows 版含む)で開いても文字化けしない一般的な選択
4. **生成方式**: クライアント側で生成しダウンロード。Firestore から取得済みのデータを CSV 文字列に変換し Blob ダウンロードする。サーバー(Cloud Functions)は不要

## 前提(仮定 — 誤りがあれば指摘してください)

1. 出力対象は **現在参加中の book(`useBook()` の `bookId`)の全期間の価格記録**とする。`docs/spec-issue17.md` の底値対象期間(`bottomWindowMonths`)によるフィルタは適用しない(エクスポートは「持ち出し」目的であり、設定期間で絞ると記録が欠落するため)。既存の `usePriceRecords()`(オプションなし呼び出し)がこの用途に合致する
2. `productId` / `storeId` は生の Firestore ドキュメント ID のままでは意味を持たないため、`useProducts()` / `useStores()` で取得した名前に解決し、CSV には ID ではなく「商品名」「店舗名」列として出力する。解決できない(削除済み等で参照が見つからない)場合は空文字とする
3. 列構成(確定):`記録日時, 商品名, 店舗名, 価格, 内容量, 単位, セール, メモ`
   - `記録日時`: `recordedAt`(Timestamp)を `YYYY-MM-DD HH:mm` 形式のローカル時刻文字列に変換
   - `セール`: `isSale` を「はい」/「いいえ」の日本語表記に変換(生の `true`/`false` より可読性が高いため)
   - `メモ`: `note`(任意)。未設定時は空文字
   - 商品作成時の `categoryId` はここでは出力しない(価格記録のみが対象という確定事項に合わせ、カテゴリはマスタ扱いとしてスコープ外)
4. CSV のフィールドエスケープ(カンマ・ダブルクォート・改行を含む値、特に `メモ` や商品名)は RFC 4180 に準拠して二重引用符で囲みエスケープする。既存依存に CSV 生成ライブラリは無いため、小さな自前のエスケープ関数を実装する(新規 npm 依存を追加しない = `docs/spec.md` の Boundaries「依存パッケージの追加は Ask first」を回避できる軽量な要件のため)
5. ファイル名は `底値記録_<book名>_<YYYYMMDD>.csv` とする(book 名にファイル名として不正な文字 `/ \ : * ? " < > |` が含まれる場合は `_` に置換)
6. ダウンロードは `Blob` + `URL.createObjectURL` + 一時 `<a>` 要素の `click()` という標準的な手法を用いる(追加ライブラリ不要)
7. 価格記録が 0 件の場合でも、ヘッダー行のみの CSV をダウンロードできる(エラーにしない)
8. 認証・Firestore セキュリティルールは変更不要(既存の `usePriceRecords()` 読み取り権限をそのまま使う。エクスポート専用の Cloud Functions や新規ルールは追加しない)
9. Analytics のカスタムイベントとして `export_data`(パラメータなし。PII を含まない)を `docs/spec-issue18.md` の既存方針に倣って追加する

## Objective

### 何を作るか

設定画面から、現在参加中の底値帳(book)の価格記録全件を CSV ファイルとしてダウンロードできる機能を追加する。

### 背景・課題

Issue コメントより:「退会時の持ち出しやユーザーの安心感に寄与」。退会(アカウント削除)によりデータが完全に失われる前に、ユーザーが自分の記録した価格データを手元に残せる手段が無い。エクスポート機能があることで、退会前の持ち出しニーズに応えるとともに、「いつでもデータを取り出せる」という安心感がサービス利用の心理的ハードルを下げる。

### ユーザーストーリー

- ユーザーとして、自分が記録した価格データを CSV でダウンロードし、Excel やスプレッドシートで見返したい
- ユーザーとして、退会する前に自分の記録データを手元に保存しておきたい
- ユーザーとして、退会するかどうかに関わらず、いつでも自分のデータをバックアップしたい

## Tech Stack

追加の依存パッケージなし。既存の React 19 + Firebase(Firestore)+ ブラウザ標準 API(`Blob` / `URL.createObjectURL`)のみで実装する。

## Commands

既存: `npm run lint` / `npm run test` / `npm run build`(追加コマンドなし)

## Project Structure(追加分)

```
src/lib/csv.ts               → 汎用 CSV エスケープ・生成ユーティリティ(RFC 4180 準拠)
src/features/prices/export.ts → PriceRecord[] + Product[] + Store[] → CSV 文字列への変換、ファイル名生成、ダウンロード実行
```

`SettingsPage.tsx` にボタンを追加し、`export.ts` の関数を呼び出す。

## Code Style

親仕様 `docs/spec.md` から変更なし。既存の `features/*/api.ts` の関数分割スタイル(Firestore アクセスは `api.ts` 側、UI ロジックは呼び出し側)を踏襲し、CSV 生成は純粋関数として `export.ts` に切り出す(テストしやすくするため)。

## Testing Strategy

| レベル | 対象 |
|---|---|
| 単体 | `src/lib/csv.ts` のエスケープ関数(カンマ・ダブルクォート・改行を含む値が正しくクォートされること) |
| 単体 | `src/features/prices/export.ts` の CSV 生成関数(`PriceRecord[]` → 期待するヘッダー・行が生成されること、`productId`/`storeId` が名前に解決されること、未解決時に空文字になること、0 件時にヘッダーのみになること) |
| 手動 | 設定画面のボタンから実際にダウンロードし、Excel(または Numbers/スプレッドシート)で文字化けなく開けることを確認 |

## Boundaries

- **Always**
  - 新規 npm 依存を追加しない(小さな自前実装で完結させる)
  - エクスポートは現在ログイン中ユーザーが参加している book のデータのみを対象とする(他ユーザーの book データにアクセスしない)
  - Analytics イベントに PII(価格記録の内容・商品名等)を含めない
- **Ask first**
  - 対象データ範囲をマスタ(商品・カテゴリ・店舗)や複数 book 対応に広げる場合
  - CSV 以外の形式(JSON 等)への対応を追加する場合
- **Never**
  - サーバー(Cloud Functions)にエクスポート対象データをログ出力しない(クライアント完結のため該当リスクは低いが念のため明記)

## Success Criteria

- [ ] 設定画面に「データをエクスポート」ボタンが表示される
- [ ] ボタン押下で現在の book の価格記録全件(期間フィルタなし)が CSV としてダウンロードされる
- [ ] CSV は UTF-8(BOM 付き)で、Excel で開いても日本語が文字化けしない
- [ ] 商品名・店舗名が ID ではなく名前で出力される
- [ ] カンマ・ダブルクォート・改行を含むメモ・商品名が正しくエスケープされ、列がずれない
- [ ] 価格記録が 0 件でもエラーにならずヘッダーのみの CSV がダウンロードされる
- [ ] `export_data` の Analytics イベントが記録される
- [ ] 既存テスト(`npm run test` 等)がすべて通る
