# Spec レビュー結果: 底値帳 Web アプリ「そこねこ」

> 対象: `docs/spec.md`(Status: Draft / 2026-07-12 版)
> レビュー日: 2026-07-12
> 位置づけ: spec-driven development Phase 1 (Specify) のレビュー記録。
> High の指摘を spec.md に反映のうえ、再レビュー → 承認へ進むこと。

## サマリー

仕様全体はよく整理されており、データモデル・技術選定・Boundaries の方針は妥当。
ただし **表示・UX の根幹に関わる未定義事項が 4 件(High)** あり、承認前に仕様への反映が必要。

| 重要度 | 件数 | 対応方針 |
|---|---|---|
| High | 4 | 承認前に spec.md へ反映必須 |
| Medium | 4 | 実装開始までに決定(手戻り防止) |
| Low | 4 | 任意。実装時に判断でも可 |

---

## High: 実装前に解消すべき

### H-1. 価格記録の編集・削除が仕様にない

- 「3 タップ + 数値入力」の最速入力を最優先にする以上、入力ミスは必ず起きる
- しかし誤記録の修正・削除は、ユーザーストーリー・画面構成・Success Criteria のいずれにも登場しない
- 誤った底値が一覧に残り続けると「この価格は買いか?」という判断の信頼性が崩れる
- **対応**: 価格記録の編集・削除を MVP スコープ(ユーザーストーリー・画面・Success Criteria)に明記する

### H-2. 参照整合性(削除時の挙動)が未定義

- Firestore には外部キー制約がないため、設定画面から店舗・カテゴリを削除すると
  `priceRecords.storeId` / `products.categoryId` が宙に浮く(孤児レコード)
- 商品削除時に配下の priceRecords をどうするかも未定義
- **対応**: 方針を仕様に定める。候補:
  - 参照されている間は削除禁止
  - 論理削除(`deletedAt` フラグ)
  - 孤児レコードは「不明な店舗」等として表示

### H-3. 底値算出に特売価格を含めるかが未定義

- 「特売と通常を区別して記録・確認」とあるが、ホームに表示する「底値」が
  特売込みの最安値か、通常価格の最安値か(あるいは両方表示か)が決まっていない
- アプリの中心的な表示仕様であり、レビュー段階で確定させるべき
- **推奨**: 底値 = 特売込みの最安値とし、特売由来の底値にはバッジ等で区別表示

### H-4. 初回 book 自動作成の競合と「自分の book の見つけ方」

- 「初回ログイン時に book を自動作成」は、複数端末での同時初回ログインや
  オフライン起動時に book が二重作成されるリスクがある
- 「自分の book をどう検索するか」(`memberUids array-contains uid` クエリ等)も未仕様
- **推奨**: MVP では `bookId = uid` の決定的 ID にする。最も簡単で確実、
  将来の複数 book 化とも矛盾しない

---

## Medium: 実装開始までに決めておくと手戻りが減る

### M-1. 単位換算の仕様が薄い

- `lib/` に「単位換算」があり Testing Strategy にも登場するが、
  データモデル上は `unit: string`(baseUnit と「同じ単位系」)としか書かれていない
- 米 5kg(baseUnit: g)や洗剤 1.2L(baseUnit: ml)は普通に発生する
- **対応**:
  - 許容する単位ペア(kg→g、L→ml)と換算表を仕様に明記する
  - 商品のカテゴリを後から変更した場合(g 系 → 個 系など)、
    既存記録の単位と新 baseUnit が不整合になるケースの扱いを定義する

### M-2. TanStack Query + onSnapshot の役割分担が曖昧

- 両者はどちらも「サーバ状態のキャッシュ」を担うため、併用すると真実のソースが二重になりがち
- リアルタイム購読が主なら、onSnapshot + 薄い自前フックで足りる可能性がある
- **対応**: 採用理由と使い分け(または片方への一本化)を 1 行でよいので仕様に明記する

### M-3. セキュリティルールの詳細が「memberUids のみ読み書き可」だけ

- 書き込み権限があるメンバーは `memberUids` 自体を書き換えられる
- 将来の共有を見据えた設計にしている以上、以下をルール仕様(Boundaries の Always)に含めることを推奨:
  - `ownerUid` は不変
  - `memberUids` の変更は owner のみ
  - book 作成時は `ownerUid == request.auth.uid` を検証
- 価格の正数チェック等のバリデーションをルール側でも行うか(クライアントのみか)も決めておくと、
  ルールテストの範囲が明確になる

### M-4. オフライン閲覧の前提条件が Success Criteria に表れていない

- Firestore のオフライン永続化は「一度オンラインで読んだデータ」しかキャッシュしない
- 「機内モードで記録 → 同期」は検証項目にあるが、前提 4 の「オフラインで閲覧」の
  受け入れ条件が未定義
- **対応**: 「オンラインで一度開いた後なら機内モードで底値一覧が見られる」等を
  Success Criteria / 手動 E2E チェックリストに明記する
- 実装時の注意: 複数タブでの永続化には `persistentMultipleTabManager` が必要

---

## Low: 好みの範囲(任意)

- **L-1. パスワードリセット**: メール/パスワード認証のリセットフローが画面構成・Success Criteria にない。知り合いに配るなら必要になる
- **L-2. Open Questions の体裁**: 「なし」の直後に箇条書きが続き紛らわしい。「解決済み事項」等の見出しに分けると誤読を防げる
- **L-3. PWA の更新戦略**: autoUpdate か prompt か未定義。オフライン重視アプリでは「更新があります」プロンプト方式のほうが事故が少ない
- **L-4. quantity の意味**: 5 箱パック × 160 組のような複合内容量は総量を入力する、等を入力 UI の仕様として一言添える

---

## 良い点

- 1 ユーザー = 1 book としつつ `books/{bookId}` + メンバー配列で将来の共有に備える設計は、
  MVP コストと拡張性のバランスが良い
- クライアント側集計の割り切り(数百〜数千レコード規模)と、スケール時に非正規化を検討する段階論も妥当
- Boundaries(Always / Ask first / Never)が具体的で、エージェント駆動開発の指針として機能する内容

# 再レビュー(2026-07-12 / 1 回目の反映後)

初回指摘の High 4・Medium 4・Low 4 は spec.md に反映済みであることを確認。
反映内容に対する新規指摘は以下のとおり。

## M-5. 「setDoc(冪等)で行い、既存 book があれば上書きしない」は自己矛盾

- setDoc はドキュメントが存在すれば上書きするため、2 台目の端末で初回ログインした際に
  ユーザー変更済みの `name` / `bottomWindowMonths` がデフォルト値で上書きされるバグになる
- **対応(反映済み)**: `getDoc` で存在確認し、存在しない場合のみ作成する方式に変更。
  初期化はトランザクションで create 専用にし、ルール側は update 時に
  `ownerUid` / `createdAt` の不変を検証。シードカテゴリは決定的 ID(`food` 等)で冪等化

## Low(任意)

- **L-5**: 商品削除のバッチは Firestore の 1 バッチ 500 書き込み上限に注意。
  分割バッチの注記を追加(反映済み)
- **L-6**: 特売記録しかない商品は「通常価格のみの底値」が算出不能。
  「該当なし(—)」表示の注記を追加(反映済み)
- **L-7**: 店舗・カテゴリの削除禁止は UI 側のみの制御でルールでは強制しない。
  意図的な割り切りである旨を明記(反映済み)
